# Testing And Ops Memory

## Local Validation Baseline

Use `LOCAL_TESTING.md` as the main runbook for local end-to-end validation.

## Known Local Stack Shape

- Docker Compose orchestrates the local environment.
- PostgreSQL initialization reads the `POSTGRES_DATABASES` Docker Compose value, creates dedicated `driver_location`, `simulation_service`, and `optimization_service` databases, and now provisions each service table set from `postgres-init/01_create_service_databases.sh`; Hibernate DDL auto-creation is disabled for those services. Existing PostgreSQL data directories require the initializer to be run manually when a database or table set is added.
- MySQL initialization provisions the shared `ride_requests_db` schema from `mysql-init/02_schema.sql`; if an existing volume predates schema changes, rerun that script manually before restarting `ride-request`, `outbox-publisher-service`, and `dashboard-service`.
- `simulation-service` and `optimization-service` are built and started by the root Docker Compose stack with Config Server, Eureka, Kafka, Redis, and PostgreSQL dependencies.
- Core checks include gateway reachability, config endpoint reachability, Kafka topic behavior, Redis data presence, and MySQL persistence.
- The outbox flow is important enough to verify explicitly, including dashboard WebSocket delivery and acknowledgement handling.

## Testing Heuristics

- Prefer unit tests for internal logic changes.
- Add integration tests when a change crosses Kafka, RabbitMQ, Redis, or MySQL boundaries.
- When event envelopes or status enums change, verify serialization and persistence together.
- For optimization work, add unit tests for snapshot assembly, constraint scoring, and baseline-vs-optimized comparison before adding solver-heavy integration coverage.
- `optimization-service` follows the repository's service-contract convention: injectable service APIs live under `service` subpackages and Spring beans live under `service/serviceimpl` with `Impl` suffixes.
- Keep builders out of `optimization-service` service implementations. Use focused mapper classes for driver snapshots, ride snapshots, simulation state, optimization snapshots, comparison metrics, distance matrices, Timefold planning models, and persisted or published results.
- For optimization integration work, verify Kafka contract handling, Redis GEO reads, and PostgreSQL result persistence together because stale snapshot bugs can hide between those boundaries.
- On the current local JDK 25 setup, JaCoCo 0.8.12 and Mockito inline mocking can fail before assertions run (inline mock maker cannot attach the Byte Buddy agent on JDK 25). When validating targeted unit tests locally, prefer `-Djacoco.skip=true` and keep `mock-maker-subclass` test overrides in mind.
- Only `driver-location-generator` and `ride-request` carry the `src/test/resources/mockito-extensions/org.mockito.plugins.MockMaker` (`mock-maker-subclass`) override. `simulation-service` and `optimization-service` currently pass without it because their tests avoid inline mocking; adding a final-class or inline mock to those services' tests will fail on JDK 25 until the same override is added.
- Verified 2026-06-24 with `mvn -pl simulation-service,optimization-service,driver-location-generator -am clean test -Djacoco.skip=true`: BUILD SUCCESS — simulation-service 13 (7 classes), optimization-service 15 (9 classes), driver-location-generator 42 (14 classes) tests, all passing (70 total). Note: stale surefire `.txt` reports for deleted test classes can survive across runs and inflate manual aggregation; trust a `clean` run or the reactor summary, not raw report counts.

## Runtime Happy-Path Drive (verified 2026-06-24)

- Docker Desktop is allocated only ~4 GiB locally; the full 14-container Compose stack uses ~3.2 GiB+ and OOM-kills containers with exit code 137 (observed for `discovery`, `kafka`, `mysql`, and even `simulation-service` mid-run). Raise Docker memory (≈8 GiB) or bring up a service subset before expecting a stable full-stack run. When a service shows persistent `Bootstrap broker kafka:9092 ... disconnected` or `UnknownHostException: discovery`, check `docker compose ps -a` for an `Exited (137)` infra container first — the app is usually fine, its dependency was OOM-killed.
- The three sim/opt/generator services set `server.port=${PORT:0}` (random ephemeral port) and register with Eureka. To drive their REST controllers: either go through the gateway at `http://localhost:8010/<lowercase-service-id>/...` (requires `discovery` up AND the gateway's Eureka registry refreshed — a long discovery outage leaves the gateway 404ing until it re-fetches), or bypass the gateway by reading the actual port from `docker compose logs <svc> | grep "Tomcat started on port"` and running `docker compose exec <svc> curl localhost:<port>/...`.
- Verified happy path: `POST /api/v1/simulations/{CONCERT_RAIN|AIRPORT_RUSH}/start?requestedBy=...` (202 + simulationRunId) → simulation persists `simulation_run` RUNNING, generates ~30 `driver_profile`, publishes `driver.generated` (30), `simulation.ride.requested` (~112), `scenario.started`, `simulation.metrics.updated`. `driver-location-generator` consumes `driver.generated`, seeds Redis GEO key `vehicle_location` + `driver:geo-state:*` (one per driver), and moves drivers every 5s publishing `driver.location.updated` (GEOPOS changes across a tick). `POST /api/v1/simulations/{id}/stop` recomputes the forecast and persists a `simulation_result` with run → COMPLETED (survives a mid-run service restart by rebuilding context from Postgres). `POST /api/v1/optimizations/run?simulationRunId=...&scenario=CONCERT_RAIN&trigger=MANUAL` (202) → `optimization_run` with `solver_status=HEURISTIC_BEST_SOLUTION_FOUND`, plus `optimization_result` and `baseline_vs_optimized_metrics` rows. Note: `optimization_run` has no `status` column — its state column is `solver_status`.

## Lean Local Stack + Simulating Omitted Services (2026-06-24)

- A memory-safe lean subset for sim/opt/generator work is: `kafka redis postgres centralized-config discovery` + the three services, plus `rabbitmq` (centralized-config `depends_on` it) and `gateway`. `mysql` and the MySQL-backed services (`ride-request`, `outbox-publisher-service`, `dashboard-service`, `location-saver`, `driver-location-streamer`) are NOT needed by the trio — stop them to keep ~14 containers from OOMing at 4 GiB. This subset runs at ~66% of the 4 GiB allocation and keeps `discovery` alive.
- The omitted dispatch/movement services can be SIMULATED purely over Kafka because both the simulation and generator consumers deserialize with a plain Jackson `ObjectMapper.readValue(data, fixedType)` per listener (no `__TypeId__` header, no trusted-packages, `FAIL_ON_UNKNOWN_PROPERTIES=false`) — so `kafka-console-producer` JSON works directly. `scripts/simulate-omitted-services.sh` automates this: publish `driver.notified` (drives simulation's DriverDecisionEngine → `simulation.driver.accepted/refused`, moves `acceptedRides`/`acceptanceProbability` off zero), `ride.assigned` (assignment metric), and `driver.command.move-to-pickup`/`start-trip` (drives generator non-idle movement → `driver.reached.pickup`/`driver.reached.destination`). Drivers must belong to the active run (`driver_profile.external_driver_id WHERE simulation_run_id=...`) or simulation ignores the notification ("no active scenario").
- GATEWAY ROUTING (fixed 2026-06-24): `gateway` runs Spring Cloud 2025.1.1 / Boot 4.0.2 / spring-cloud-gateway 5.0.1. Two bugs blocked all routing (every `/{service-id}/**` returned 404): (1) it depended on `spring-cloud-starter-gateway-server-webmvc`, which has NO discovery-locator feature at all — swapped to the reactive `spring-cloud-starter-gateway-server-webflux` in `gateway/pom.xml`; (2) on 5.x the discovery-locator property prefix moved to `spring.cloud.gateway.server.webflux.discovery.locator.{enabled,lower-case-service-id}` — the legacy `spring.cloud.gateway.discovery.locator.*` in `application.properties` was inert. Also set `eureka.client.fetch-registry=true` (was false; the locator builds routes from the fetched registry). With all three in place, `POST http://localhost:8010/simulation-service/...` and `/optimization-service/...` route correctly (lowercase service id). NOTE: after any `discovery` restart the gateway needs ~30-60s (Eureka fetch interval) before routes repopulate; expect transient 404s in that window. Direct fallback still works: read the random `${PORT:0}` Tomcat port from `docker compose logs <svc> | grep "Tomcat started on port"` and `docker compose exec <svc> curl localhost:<port>/...`.
- CAUTION: `docker compose up -d --build gateway` recreated the `discovery` dependency too (wiping Eureka's in-memory registry), which then needs the services to re-register and the gateway to re-fetch. If you only mean to rebuild one service, expect dependency containers to be recreated and plan for the re-registration lag.

## Generator Ride-Decision Consumption (implemented 2026-06-24)

- `driver-location-generator` now has `RideDecisionListener` consuming `simulation.driver.accepted` / `simulation.driver.refused` / `ride.cancelled` (new topic/group/listener keys in `driver-location-generator.properties`; new factory beans in its `KafkaListenerConfiguration`). Engine methods: `acceptRide` (→ `moveToPickup`, only if driver has no `activeRideId`), `declineRide` (log-only no-op), `cancelRide` (frees driver to `IDLE`+available via `DriverMovementEngineMapper.toFreedDriverState`, only when `activeRideId` is non-null and matches the event `rideId`; emits `driver.available`). `startTrip` is now guarded to ignore a command whose `rideId` differs from the driver's `activeRideId`.
- Contract change: simulation `DriverAcceptedEvent` gained `pickupLatitude/pickupLongitude/expectedFare` (populated from `DriverNotifiedEvent` in `RideResponseMapper`); generator `DriverLocationUpdatedEvent` gained `status` + `activeRideId`. All additive — consumers deserialize leniently.
- Verified with `mvn -pl driver-location-generator,simulation-service -am clean test -Djacoco.skip=true`: generator 52, simulation 14, all passing. New tests: generator `RideDecisionListenerTest` + decision cases in `DriverMovementEngineTest`; simulation `RideResponseMapperTest`.
- STILL DORMANT until producers exist: nothing publishes `ride.cancelled` or `driver.command.start-trip` yet, so end-to-end cancel/trip needs a producer (simulation passenger-cancellation emitter, or a ride-request bridge) — inject via `kafka-console-producer` / `scripts/simulate-omitted-services.sh` to exercise the generator paths meanwhile.
- Availability-aware dispatch matching (final design 2026-06-24): availability is owned by `simulation-service`, NOT the generator. `simulation-service.service.serviceimpl.DriverAvailabilityRegistryImpl` maintains a Redis SET `available_drivers` of driver IDs (key `simulation-service.redis.available-drivers-key`, default `available_drivers`): `markAvailable` (SADD) on driver-generated (`PopulationEngineImpl`), cancel and reached-destination (`RideResponseListener`); `markBusy` (SREM) on accept. `ride-request` `RidersSearchServiceImpl` runs GEORADIUS on `vehicle_location` (constant `VEHICLE_LOCATION`) and filters candidates by membership in the `available_drivers` SET (`opsForSet().members(AVAILABLE_DRIVERS_KEY)`), then limits to 5 — so busy drivers are skipped and freed drivers re-qualify. The generator no longer touches any availability index (the earlier generator-owned `available_drivers` GEO set + `redis.available-geo-key` were reverted). `vehicle_location` is a Redis GEO set (position-only — you CANNOT store status on a member), holds ALL drivers (generator + location-saver), and feeds optimization's full-fleet view. Verified: generator 52, simulation 17 (incl. `DriverAvailabilityRegistryImplTest`), ride-request `RidersSearchServiceImplTest` 9 — all passing. Caveat: only the simulation populates `available_drivers`, so a real (location-saver-fed, no-simulation) deployment would have an empty set → consider a `vehicle_location` fallback when the set is empty. The ride-request Testcontainers ITs were updated to seed BOTH `vehicle_location` (positions) and `available_drivers` (SADD) and compile, but were not run locally (need Docker).

## Operational Heuristics

- If a change alters startup config, inspect Spring properties, centralized config, and deployment values together.
- MySQL table names are canonicalized to lowercase. The driver dispatch projection is `driver`.
- `simulation-service` now follows the repo's `service` plus `serviceimpl` split and reads runtime properties from `centralized-config/centralized-configuration/simulation-service.properties`; when debugging bean wiring or missing config, check the interface injection point and that config file together.
- In `simulation-service`, entity state transitions such as run completion and metrics status updates should be routed through mapper helpers rather than applied inline in services; check `mapper/` first when a status mutation looks wrong.
- If a change alters networking or routing, inspect gateway, discovery, Helm, and Terraform impact together.
- If a change alters local workflows, update `LOCAL_TESTING.md`.
- If optimization recommendations begin driving actual movement commands, inspect `driver-location-generator`, centralized config, and local replay steps together before treating the flow as operationally complete.
- `dashboard-service` still runs with `spring.jpa.hibernate.ddl-auto=none`, but now relies on MySQL-specific startup SQL init for cross-service read-model columns. Keep `centralized-config/centralized-configuration/dashboard-service.properties` and `dashboard-service/src/main/resources/schema-mysql.sql` aligned whenever dashboard queries add new columns.
